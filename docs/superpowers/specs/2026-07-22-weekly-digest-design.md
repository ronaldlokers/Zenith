# Weekly accountability digest — design

**Date:** 2026-07-22
**Status:** approved

## Problem

Zenith surfaces momentum/funnel stats only when the user opens the app. A
weekly, proactive recap — reaching the user via the existing notification
center + push — reinforces the habit and, crucially, surfaces **stalled**
applications (in-flight but gone quiet), which no current view emphasizes.
2026 job-search burnout is a documented pain point with no strong incumbent
solution; a gentle weekly nudge (not streaks/gamification) is genuine
whitespace.

## Surface (decided)

A weekly `weekly_digest` notification (center entry + push), generated by a new
weekly cron. Chosen over an in-app-only card because the accountability value is
*proactive*.

## i18n constraint (decided approach)

The digest is generated prose ("this week: 4 added · 2 advanced · 3 stalled"),
unlike existing notifications that reuse raw DB content as their title. The
worker has no react-i18next and there is currently **no server-side user
locale** (UI language lives in `localStorage:zenith_lang` only). Per the "all
user-facing strings externalized, strict en/nl parity" rule, the digest can't be
English-in-worker.

**Decision: persist the user's UI language server-side** so the worker localizes
push + center fully. This is a clean, reusable enabler and is built as its own
PR first.

## Decomposition — two PRs

### PR A — persist user UI language server-side

- **Migration:** `ALTER TABLE "user" ADD COLUMN locale TEXT;` (mirrors the
  `twoFactorEnabled` precedent, migration 0030). Nullable; worker treats NULL as
  `'en'`.
- **Endpoint:** `PUT /api/preferences/locale` (session-auth) — body `{locale}`,
  validated against `['en','nl']`, writes `"user".locale` for the session user.
- **Frontend:** the settings language `<select>` `onChange` already calls
  `i18n.changeLanguage`; also POST the new locale (fire-and-forget, non-blocking;
  a failure only means the server copy lags, UI still switches). Optionally
  backfill on load if the server has NULL and localStorage has a value.
- **Test:** PUT valid locale → persisted; invalid locale → 400.

### PR B — the weekly digest

- **Migration:** rebuild the `notifications` CHECK to add `'weekly_digest'`
  (SQLite can't alter a CHECK in place; same rebuild pattern as 0041).
- **Worker i18n helper** (`worker/i18n.ts`): imports `src/locales/{en,nl}.json`,
  resolves a dotted key for a given locale, interpolates i18next-style `{{var}}`
  placeholders. Keeps digest strings in the locale files (parity enforced), lets
  the worker localize. Falls back to `en` for unknown locale/missing key.
- **Weekly cron:** add `"0 8 * * 1"` (Monday 08:00 UTC) to `wrangler.jsonc`
  crons. The `scheduled` handler branches on `event.cron` and calls
  `generateWeeklyDigest(env)`.
- **`worker/digest.ts`** — `generateWeeklyDigest(env)`: for each user, compute
  the counts below; if all zero, **skip** (no empty digest). Read `user.locale`
  (NULL → `en`), build localized `title`/`body` via the i18n helper, insert a
  `weekly_digest` notification (`dedup_key = 'weekly_digest:' || <week-start>`,
  `link = '/'`) and push it. Idempotent per week via dedup_key.
- **`src/types.ts`:** add `'weekly_digest'` to the `AppNotification.type` union.
- **Locale strings:** `digest.title` and `digest.body` (with `{{added}}`,
  `{{advanced}}`, `{{stalled}}`) in en.json + nl.json.

### Metrics (PR B)

Computed per user in SQL:
- **added** — applications with `created_at >= date('now','-7 days')`.
- **advanced** — distinct applications with a *forward* `status_history` move in
  the last 7 days (`to_status` rank > `from_status` rank; rank via a CASE over
  the stage ladder interested<applied<screening<interview<offer).
- **stalled** — applications with `status IN ('applied','screening','interview')`
  (active, non-terminal, past the interested stage), `created_at` older than 14
  days, and **no** `status_history` change and **no** interaction in the last 14
  days.

The digest is skipped entirely when `added + advanced + stalled = 0` (avoids a
demotivating empty recap).

## Files

**PR A:** `migrations/0042_user_locale.sql`, `worker/index.ts` (or a prefs route
module) for the endpoint, `src/settings/index.tsx` (persist on change),
`src/api.ts` (client call), a worker test.

**PR B:** `migrations/0043_notification_weekly_digest.sql`, `worker/i18n.ts`,
`worker/digest.ts`, `worker/index.ts` (cron branch + wiring), `wrangler.jsonc`
(cron), `src/types.ts`, `src/locales/{en,nl}.json`, `worker/digest` test.

## Test plan

- **PR A:** endpoint persists/validates locale.
- **PR B:** seed applications/status_history/interactions for the seed user →
  `generateWeeklyDigest` → assert one `weekly_digest` row with correct counts in
  the localized body; re-run → no duplicate (dedup); zero-activity user → no row;
  a `locale='nl'` user → Dutch body. Gates: tsc, build, oxlint, vitest, en/nl
  parity.

## Out of scope (YAGNI)

- In-app "Your week" card (notification + landing on Overview is enough for v1).
- Configurable cadence / digest day (fixed Monday 08:00 UTC).
- A dedicated digest detail view (links to Overview, which already shows stats).
- Per-type notification opt-out (parity with existing notification types).
